# Admin UI Rules

管理后台是任务型、数据型、操作型界面。默认目标是高效、准确、稳定、低误操作，而不是制造营销页式视觉冲击。

## 默认技术栈

- UI 组件：`shadcn/ui`
- 样式系统：`Tailwind CSS`
- 图标：`lucide-react`
- 复杂表格：优先使用 `TanStack Table` 或项目已有表格方案
- 表单：优先使用 schema 驱动，例如 Zod + React Hook Form
- 数据请求：优先使用项目已有 typed request；React 项目可叠加 TanStack Query

## shadcn MCP 使用规则

- 开始管理端 UI 任务前，优先通过 shadcn MCP 查询相关组件、blocks 和安装方式。
- 组件来源优先级：
  1. 项目已有业务组件和布局组件
  2. 项目已有 shadcn 组件
  3. shadcn 官方组件
  4. shadcn registry / blocks
  5. 基于现有 token 和组件风格封装的新组件
- 能用 shadcn 组件解决时，不要手写新的 Button、Input、Dialog、Table、Select、Dropdown 等基础组件。
- 安装组件必须使用 shadcn MCP 返回的安装方式或 shadcn 官方 CLI 命令。
- 修改 shadcn 组件时，只做项目级 token、variant、可访问性、布局适配；不要破坏组件的通用结构。
- 新增自定义组件必须说明它解决的复用问题，并保持与 shadcn 组件 API、Tailwind token、项目命名风格一致。

## Tailwind 规则

- 颜色、字体、间距、圆角、阴影必须来自项目主题 token 或 shadcn 主题变量。
- 不允许在页面中随机新增 HEX、RGB、HSL 色值；确需新增颜色时，先补充主题 token。
- 间距默认使用 4/8px 系统，例如 `gap-2`、`gap-4`、`p-4`、`px-6`。
- 后台页面不要使用营销页式大字号、大留白和大面积居中排版，例如无理由的 `text-5xl`、`py-24`。
- 圆角默认不超过 `8px`，除非项目设计系统已有更明确规则。
- 主色只用于关键操作、选中状态、焦点状态和少量强调，不要让页面被单一高饱和色占满。
- 默认避免大面积渐变、玻璃拟态、装饰背景、夸张阴影和过度动效。

## Lucide 图标规则

- React 项目默认从 `lucide-react` 按需导入图标。
- 后台界面图标尺寸默认使用 `16px` 或 `20px`。
- 图标线宽保持一致，默认使用 Lucide stroke 风格。
- 有现成 Lucide 图标时，不要手写临时 SVG。
- 图标必须服务识别和操作，不要为了装饰堆叠图标。
- 只有图标的按钮必须提供 `aria-label`，并在需要时提供 tooltip。

## 后台布局

- 默认结构：左侧导航、顶部栏、主内容区。
- 主内容区应包含稳定的页面标题、主要操作、状态反馈和必要的筛选入口。
- 面包屑用于层级较深的管理界面，不要替代页面标题。
- 列表页默认包含搜索、筛选、表格或列表、分页、批量操作、空状态。
- 详情页默认包含基础信息、状态、关联数据、操作记录和危险操作区。
- 创建和编辑页优先使用完整表单页或 `Sheet`；字段较多时不要塞进小弹窗。
- 删除、禁用、重置密钥、清空数据等危险操作必须二次确认。

## 数据列表和表格

- 表格必须支持 loading、empty、error、success 状态。
- 表头文案必须短、明确、可扫描。
- 数字、金额、百分比、时间、状态必须格式化后展示。
- 状态字段优先使用 `Badge` 或稳定的状态样式，不要只依赖颜色表达含义。
- 行操作超过两个时，使用 `DropdownMenu` 收纳次要操作。
- 批量操作必须显示选中数量和影响范围。
- 筛选条件必须可回显、可清除，复杂筛选应提供重置入口。
- 分页、排序、搜索和筛选状态应进入 URL 或可恢复状态，避免刷新后丢失上下文。
- 表格列很多时，优先固定关键列、收纳次要字段或提供列显示控制，不要让页面横向失控。

## 表单和操作反馈

- 表单字段必须有明确 label，placeholder 不能替代 label。
- 必填、格式、权限、业务规则错误必须显示在对应字段附近。
- 提交按钮必须有 loading/disabled 状态，避免重复提交。
- 保存、删除、禁用、重置、导入、导出等操作必须有成功和失败反馈。
- 长任务必须显示进度、队列状态或明确的等待反馈。
- 破坏性操作使用 destructive variant，并说明后果。
- 权限不足时不要只隐藏关键区域；需要展示不可用状态、替代入口或明确原因时，应给出用户能理解的反馈。

## 移动端适配

管理后台必须考虑移动端，但不要求所有复杂任务在手机上达到桌面效率。按任务分级处理：

- 必须移动端可用：登录、查看关键状态、审批、启停开关、查看详情、处理告警、单条轻量操作。
- 移动端降级可用：列表浏览、搜索、基础筛选、单条编辑、简单表单提交。
- 建议桌面完成：批量导入导出、复杂报表、多条件高级筛选、批量编辑、复杂权限配置、字段很多的长表单。

移动端实现要求：

- 桌面表格在移动端优先转为卡片列表、折叠字段或横向可控滚动。
- 关键操作按钮点击目标不小于 `40px`。
- 筛选区在移动端优先使用 `Sheet` 或折叠面板。
- 批量操作在移动端必须清楚显示选中数量和操作后果。
- 侧边导航在移动端使用抽屉或底部导航，不要挤压主内容。
- 表单字段在移动端单列展示，错误信息不能遮挡输入区域。

## 可访问性和可操作性

- 所有交互控件必须可键盘访问。
- 焦点状态必须可见。
- 颜色对比度必须满足常规可读性要求。
- 不要只用颜色传达状态；状态必须有文字、图标或语义标签辅助。
- 弹窗、抽屉、菜单必须有合理的焦点管理和关闭方式。
- Toast 只能作为反馈，不应承载必须长期阅读的关键信息。
- 确认弹窗文案必须说明操作对象和后果。

## 禁止项

- 不要把管理后台做成营销页、品牌首页或作品集页面。
- 不要使用大 hero、装饰性大图、无意义动效和花哨背景。
- 不要为了视觉效果牺牲数据密度、可扫描性和操作效率。
- 不要随机创造颜色、间距、阴影、圆角、图标风格或组件风格。
- 不要在用户可见界面写入设计说明、实现说明、组件说明、内部流程或作者备注。
- 不要隐藏错误、权限不足、加载失败和空数据状态。
